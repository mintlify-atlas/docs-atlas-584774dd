The keeper service monitors the Settlement contract for token balances, posts swap orders to CoW Protocol, and executes the on-chain drip transaction. This guide covers running the keeper directly via Node.js.

<Note>
For containerized deployment, see [Docker Deployment](/operations/docker).
</Note>

## Quick Start

Run the keeper with minimal configuration:

```bash
source .env

yarn ts-node index.ts \
  --network mainnet \
  --module 0x8429bf5bad17925c364c8c30a3c2a1e0d76602d2
```

<Warning>
Ensure `PRIVATE_KEY` environment variable is set before running.
</Warning>

## Execution Flow

When you run the keeper, it executes the following workflow:

<Steps>
  <Step title="Configuration Loading">
    Loads configuration from CLI options and environment variables
    
    **From `ts/utils/readConfig.ts:99-118`:**
    ```typescript
    const [receiver, wrappedNativeToken, vaultRelayer, gpv2Settlement, 
           keeper, appData, targetSafe, minOut] = await Promise.all([
      moduleContract.receiver(),
      moduleContract.wrappedNativeToken(),
      moduleContract.vaultRelayer(),
      moduleContract.settlement(),
      moduleContract.keeper(),
      moduleContract.appData(),
      moduleContract.targetSafe(),
      moduleContract.minOut(),
    ]);
    ```
  </Step>
  
  <Step title="Token Discovery">
    Discovers tokens held by the Settlement contract
    
    **From `ts/drip/dripItAll.ts:22`:**
    ```typescript
    const tokensToSwap = await getTokensToSwap(config, provider);
    ```
    
    See [How It Works](/how-it-works#token-discovery-process) for details.
  </Step>
  
  <Step title="Filtering">
    Filters tokens by liquidity (quotes API) and minOut threshold
    
    **From `ts/drip/getTokensToSwap.ts:88-93`:**
    ```typescript
    const minOutFiltered = quotesFiltered.filter((token) =>
      BigNumber.from(token.buyAmount).gt(config.minOut)
    );
    console.log("Total tokens after filtering by minOut:", minOutFiltered.length);
    ```
  </Step>
  
  <Step title="Order Posting">
    Posts swap orders to CoW Protocol OrderBook API
    
    **From `ts/drip/swapTokens.ts:24-32`**
    
    Console output:
    ```
    Successfully posted orders:
     [ { token: '0x...', orderId: '0x...' }, ... ]
    ```
  </Step>
  
  <Step title="Drip Transaction">
    Builds and sends the on-chain drip transaction
    
    **From `ts/drip/drip.ts:65-76`:**
    ```typescript
    const txBaseRequest: TransactionRequest = {
      from: await signerWithProvider.getAddress(),
      to: moduleContract.address,
      data: moduleContract.interface.encodeFunctionData("drip", [
        toApprove,
        toDrip,
      ]),
      ...
    };
    ```
  </Step>
  
  <Step title="Monitoring">
    Prints expected proceeds and CoW Explorer link
    
    **From `ts/drip/dripItAll.ts:74-80`:**
    ```typescript
    console.log(
      `Fee collection for chain ${config.network} initiated (${
        tokensToSwap.length
      } orders). Expecting proceeds of ${expectedUnitsFormatted} ${
        await buyTokenContract.symbol()
      }!\n\nFollow the progress at ${
        networkSpecificConfigs[config.network].explorer
      }/address/${config.gpv2Settlement}`
    );
    ```
  </Step>
</Steps>

After the drip transaction is confirmed, solvers will fill the orders.

### CoW Explorer

Visit the CoW Explorer link from the output to monitor order fills:

**Network-specific explorers (from `ts/config.ts:24-83`):**

| Network | Explorer URL |
|---------|-------------|
| Mainnet | `https://explorer.cow.fi` |
| Gnosis | `https://explorer.cow.fi/gc` |
| Arbitrum | `https://explorer.cow.fi/arb1` |
| Base | `https://explorer.cow.fi/base` |
| Sepolia | `https://explorer.cow.fi/sepolia` |

### Check Order Status

Orders appear in the Settlement contract's address page with status:
- **Open**: Waiting for solver
- **Filled**: Successfully executed
- **Expired**: Past `validTo` timestamp
- **Cancelled**: Explicitly cancelled

### Verify Final Proceeds

Check the receiver address balance:

```bash
cast call $WRAPPED_NATIVE_TOKEN \
  "balanceOf(address)(uint256)" \
  $RECEIVER \
  --rpc-url $RPC_URL
```

Or use block explorer to view the receiver's wrapped token balance.

## Common Scenarios

### No Tokens to Swap

If no tokens pass the filters:

```bash
ETH to wrap: 0.0
Total tokens pre-filter: 12
Total tokens after filtering by quotes api: 5
Total tokens after filtering by minOut: 0
No tokens to swap, skipping approvals and drip
```

<Note>
This is normal if:
- All token balances are below `minOut` threshold
- Tokens lack liquidity (quotes API rejects)
- Settlement contract has no fees accumulated
</Note>

### Only Native ETH to Wrap

If only native ETH is present (no token swaps):

```bash
ETH to wrap: 1.5
Total tokens pre-filter: 0
Total tokens after filtering by quotes api: 0
Total tokens after filtering by minOut: 0

Drip transaction parameters: { ... }
Estimated gas limit: 234567

Fee collection for chain mainnet initiated (0 orders). Expecting proceeds of 1.50 WETH!
```

**From `ts/drip/dripItAll.ts:46-58`:**
```typescript

    maxPriorityFeePerGas: config.maxPriorityFeePerGas,
```

### Failed Order Posting

Some orders may fail to post:

```bash
Successfully posted orders:
 [ { token: '0x...', orderId: '0x...' } ]

Failed posting orders:
 [ "Error: Insufficient liquidity for token 0x...",
   "Error: Invalid quote response" ]
```

Only successfully posted orders are included in the drip transaction.

## Troubleshooting

<Accordion title="Error: Missing env var PRIVATE_KEY">
  **Cause:** `PRIVATE_KEY` environment variable not set.
  
  **Solution:**
  ```bash
  export PRIVATE_KEY=0xYourPrivateKeyHere
  ```
  
  Or add to `.env` and run `source .env`.
</Accordion>

<Accordion title="Error: The provided private key belongs to X, which doesn't match the keeper (Y)">
  **Cause:** Private key doesn't match the keeper address from module deployment.
  
  **Solution:** Use the correct private key for the keeper address, or redeploy the module with the correct keeper address.
  
  **From `ts/utils/readConfig.ts:121-125`**
</Accordion>

<Accordion title="Error: OnlyKeeper()">
  **Cause:** Transaction reverted because `msg.sender` is not the keeper.
  
  **Solution:** Verify the private key matches the module's keeper address.
</Accordion>

<Accordion title="Error estimating gas">
  **Cause:** Transaction simulation failed.
  
  **Common reasons:**
  - Module not enabled on Safe
  - Safe not whitelisted as solver
  - Invalid order parameters
  - Insufficient token balances
  
  **Solution:** 
  1. Verify module is enabled: `cast call $TARGET_SAFE "isModuleEnabled(address)(bool)" $MODULE_ADDRESS --rpc-url $RPC_URL`
  2. Check Safe solver whitelist status
  3. Review transaction parameters in output
</Accordion>

<Accordion title="Error: Failed to get max fee per gas from network">
  **Cause:** RPC doesn't support `eth_feeData`.
  
  **Solution:** Manually specify gas prices:
  ```bash
  --max-fee-per-gas 50 --max-priority-fee-per-gas 2
  ```
  
  **From `ts/drip/drip.ts:59-62`**
</Accordion>

<Accordion title="Orders not filled">
  **Cause:** Orders may take time to fill, or lack liquidity.
  
  **Check:**
  1. Visit CoW Explorer link from output
  2. Verify orders are "Open" (not expired)
  3. Check if buy amounts are competitive
  4. Wait for solvers to fill (can take minutes to hours)
  
  Orders expire after `nextValidTo()` (next hour boundary + 2 hours).
</Accordion>

## Automation

### Cron Job

Run the keeper periodically via cron:

```bash
# Edit crontab
crontab -e

# Add entry (runs every 6 hours)
0 */6 * * * cd /path/to/cow-fee && source .env && yarn ts-node index.ts --network mainnet --module 0x... >> /var/log/cow-fee.log 2>&1
```

### Systemd Service

Create a systemd service for automatic restarts:

```ini
[Unit]
Description=CoW Fee Keeper
After=network.target

[Service]
Type=simple
User=keeper
WorkingDirectory=/path/to/cow-fee
EnvironmentFile=/path/to/.env
ExecStart=/usr/bin/yarn ts-node index.ts --network mainnet --module 0x...
Restart=on-failure
RestartSec=300

[Install]
WantedBy=multi-user.target
```
