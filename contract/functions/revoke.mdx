## Function Signature

```solidity
function revoke(Revocation[] calldata _revocations) external onlyKeeper
```

## Overview

The `revoke()` function removes token approvals by setting allowances to zero. This is useful for security purposes, such as revoking approvals from compromised addresses or cleaning up unused approvals.

<Warning>
This function can only be called by the authorized keeper address.
</Warning>

## Parameters

<ParamField path="_revocations" type="Revocation[]">
  Array of Revocation structs specifying which tokens and spenders to revoke.
</ParamField>

## Revocation Struct

```solidity src/COWFeeModule.sol
struct Revocation {
    address token;
    address spender;
}
```

<ParamField path="token" type="address">
  Address of the ERC20 token to revoke approval for.
</ParamField>

<ParamField path="spender" type="address">
  Address of the spender whose approval should be revoked.
</ParamField>

## Behavior

The function:

1. Creates an array of interaction data for revocations
2. For each revocation, creates an approval interaction with zero allowance
3. Executes all revocations through the Settlement contract's `settle()` function

## Implementation

```solidity src/COWFeeModule.sol
/// @notice Revoke approvals for given tokens to given contracts
function revoke(Revocation[] calldata _revocations) external onlyKeeper {
    IGPv2Settlement.InteractionData[] memory revokeInteractions =
        new IGPv2Settlement.InteractionData[](_revocations.length);
    for (uint256 i = 0; i < _revocations.length;) {
        Revocation calldata revocation = _revocations[i];
        revokeInteractions[i] = IGPv2Settlement.InteractionData({
            to: revocation.token,
            value: 0,
            callData: abi.encodeCall(IERC20.approve, (revocation.spender, 0))
        });

        unchecked {
            ++i;
        }
    }
    _execInteractions(revokeInteractions);
}
```

## Zero Approval Mechanism

Revocations work by setting the allowance to zero:

```solidity
abi.encodeCall(IERC20.approve, (revocation.spender, 0))
```

This prevents the spender from transferring any tokens on behalf of the Safe.

<Note>
Some tokens (like USDT) require setting approval to zero before changing it to a non-zero value. This function facilitates that two-step approval process.
</Note>

## Settlement Contract Interaction

Revocations are executed through the Safe's interaction with the Settlement contract:

```solidity src/COWFeeModule.sol
function _execInteractions(IGPv2Settlement.InteractionData[] memory _interactions) internal {
    address[] memory tokens = new address[](0);
    uint256[] memory clearingPrices = new uint256[](0);
    IGPv2Settlement.TradeData[] memory trades = new IGPv2Settlement.TradeData[](0);
    IGPv2Settlement.InteractionData[][3] memory interactions =
        [new IGPv2Settlement.InteractionData[](0), _interactions, new IGPv2Settlement.InteractionData[](0)];
    bytes memory cd = abi.encodeCall(IGPv2Settlement.settle, (tokens, clearingPrices, trades, interactions));
    _execFromModule(address(settlement), cd);
}
```

## When to Use

Call `revoke()` when:

- A spender address is compromised or suspicious
- Cleaning up approvals that are no longer needed
- Implementing a two-step approval process (revoke to zero, then approve to new amount)
- Auditing token permissions
- Migrating to a new vault relayer address

<Warning>
Revoking the vault relayer's approval will prevent future CoW Protocol orders from executing until approval is restored.
</Warning>

## Multiple Revocations

The function accepts an array, allowing multiple revocations in a single transaction:

```javascript
const revocations = [
  {
    token: '0x6B175474E89094C44Da98b954EedeAC495271d0F', // DAI
    spender: '0x...' // Old vault relayer
  },
  {
    token: '0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48', // USDC
    spender: '0x...' // Old vault relayer
  }
];

await cowFeeModule.revoke(revocations);
```

## Error Handling

<ResponseField name="OnlyKeeper" type="error">
  Thrown if the caller is not the keeper address.
</ResponseField>

If any token revocation fails (e.g., token contract reverts), the entire transaction reverts through the Safe's `execTransactionFromModuleReturnData()` mechanism.

## Security Considerations

<Warning>
**Important:** Revoking approvals for the vault relayer will break the module's functionality until approvals are restored via `approve()` or `drip()`.
</Warning>

### Best Practices

1. **Verify addresses** before revoking to avoid accidentally breaking functionality
2. **Document revocations** in your keeper logs for audit purposes
3. **Test on testnets** before revoking approvals on mainnet
4. **Have a restoration plan** ready if you revoke critical approvals

## Gas Optimization

The function uses unchecked increment for gas efficiency:

```solidity
unchecked {
    ++i;
}
```

This is safe because the loop counter cannot overflow within practical gas limits.

## Usage Example

```javascript
// Revoke vault relayer approval for DAI
const revocations = [
  {
    token: '0x6B175474E89094C44Da98b954EedeAC495271d0F', // DAI
    spender: '0xC92E8bdf79f0507f65a392b0ab4667716BFE0110'  // Vault relayer
  }
];

await cowFeeModule.revoke(revocations);

// Later, restore approval
await cowFeeModule.approve(['0x6B175474E89094C44Da98b954EedeAC495271d0F']);
```

## Comparison with approve()

| Function | Purpose | Allowance |
|----------|---------|----------|
| `approve()` | Grant permissions | `type(uint256).max` |
| `revoke()` | Remove permissions | `0` |

Both functions use the same underlying mechanism (ERC20 `approve()`), just with different allowance values.
